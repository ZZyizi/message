# 一对一聊天端到端加密 (E2EE) 设计

## 概述

为一对一聊天实现端到端加密，保护消息在传输过程中不被 Relay 服务器窃听。采用 X25519 ECDH 密钥协商 + AES-256-GCM 对称加密方案，提供前向保密。

## 目标

- 消息 payload 在传输和存储中均为密文
- Relay 服务器无法读取消息内容
- 每次会话使用临时密钥，提供前向保密
- 复用已有的密码学原语（`derive_session_key`、`encrypt_message`、`decrypt_message`）

## 非目标

- 群组加密（后续实现）
- 多设备密钥同步（后续实现）
- Double Ratchet / Signal 协议（当前方案已满足安全需求）
- 身份密钥轮换（当前方案假设静态身份密钥不变）

## 密钥协商流程

```
Alice                                    Relay                                   Bob
  |                                        |                                      |
  |-- KeyExchange {ephemeral_pubkey, sig}->|                                      |
  |                                        |-- KeyExchange {ephemeral_pubkey, sig}->|
  |                                        |                                      |
  |<-- KeyExchange {ephemeral_pubkey, sig}-|<-- KeyExchange {ephemeral_pubkey, sig}-|
  |                                        |                                      |
  | verify sig, then:                      |                        verify sig, then:
  | SessionKey = ECDH(A_priv, B_pub)      |        SessionKey = ECDH(B_priv, A_pub)
  |                                        |                                      |
  |== KeyConfirm {encrypted("confirm")} ==> Relay ==> Bob decrypts ==============|
  |                                        |                                      |
  |<== KeyConfirm {encrypted("confirm")} ==|<== Bob confirms =====================|
  |                                        |                                      |
  |== Encrypted Message (AES-256-GCM) ===> Relay ===> Bob decrypts ==============|
```

### 步骤详解

1. Alice 打开聊天窗口 → 生成临时 X25519 密钥对
2. Alice 用静态 Ed25519 私钥对 `ephemeral_pubkey` 签名
3. Alice 发送 `KeyExchange { from, ephemeral_pubkey, signature, nonce, timestamp }` 到 Relay
4. Relay 转发给 Bob（点对点，带 `to` 字段）
5. Bob 收到后：
   - 验证 `signature` 是否由 `from` 对应的 Ed25519 公钥签署
   - 检查 `timestamp` 是否在 5 分钟内（防重放）
   - 生成自己的临时密钥对，签名后回复 `KeyExchange`
   - 调用 `derive_session_key()` 派生 SessionKey
6. Alice 收到 Bob 的 KeyExchange → 同样验证签名和时间戳 → 派生 SessionKey
7. 双方发送 `KeyConfirm` 消息确认密钥一致
8. 确认成功后开始加密通信

### HKDF info 参数

`derive_session_key()` 的 `info` 参数使用固定值：

```rust
let info = b"decentralized-im-e2ee-1to1";
```

双方必须使用相同的 `info` 值，否则派生的密钥不同。

## WebSocket 协议扩展

在 `WsMessage` 枚举中新增变体：

```rust
KeyExchange {
    from: String,                // 发送方静态 Ed25519 公钥
    to: String,                  // 接收方静态公钥（点对点路由）
    ephemeral_pubkey: String,    // 临时 X25519 公钥 (Base64)
    signature: String,           // Ed25519 签名 of ephemeral_pubkey (Base64)
    nonce: [u8; 32],            // 随机数（防重放）
    timestamp: i64,              // Unix 时间戳（秒）
}

KeyConfirm {
    from: String,                // 发送方静态公钥
    to: String,                  // 接收方静态公钥
    encrypted_confirm: String,   // encrypt("confirm", session_key) (Base64)
}
```

## 消息加解密

### 发送路径

```
明文 payload
  -> encrypt_message(payload, session_key, None)
  -> 密文 (Base64)
  -> Ed25519 签名: sign(event_id + from_pubkey + ciphertext + seq_id)
  -> 存入 messages.payload (密文) + pending_messages.payload (明文暂存)
  -> 通过 WebSocket 发送到 Relay
```

**注意：** 签名现在是对密文签名，而非明文。这确保传输内容的完整性。

### 接收路径

```
从数据库/Relay 收到密文
  -> verify_signature() 验证签名
  -> decrypt_message(ciphertext, session_key)
  -> 明文
  -> 通过 Tauri 事件发送到前端显示
```

### 数据库

- `messages.payload` 存储密文（Base64 编码），现有表结构无需修改
- `pending_messages.payload` 暂存明文，消息确认送达后清除明文并替换为密文

### 解密失败处理

- `decrypt_message()` 失败时（密钥错误/数据损坏）：前端显示"[解密失败]"
- 不崩溃，不重试解密
- 持续解密失败时重新发起 KeyExchange 协商新密钥

## 会话状态机

```
None ──> (发送 KeyExchange) ──> WaitingForPeer
WaitingForPeer ──> (收到对方 KeyExchange) ──> KeyExchanged
KeyExchanged ──> (发送 KeyConfirm) ──> WaitingForConfirm
WaitingForConfirm ──> (收到对方 KeyConfirm) ──> Active
Active ──> (收到新 KeyExchange) ──> Rekeying
```

```rust
enum SessionStatus {
    None,                          // 未开始协商
    WaitingForPeer,                // 已发送 KeyExchange，等待对方
    KeyExchanged,                  // 已收到对方 KeyExchange，已派生密钥
    WaitingForConfirm,             // 已发送 KeyConfirm，等待确认
    Active,                        // 会话已建立，可通信
    Rekeying,                      // 正在重新协商密钥
}

struct SessionState {
    status: SessionStatus,
    peer_static_pubkey: String,
    my_ephemeral_privkey: [u8; 32],     // 临时私钥（不写入磁盘）
    peer_ephemeral_pubkey: Option<[u8; 32]>,
    session_key: Option<[u8; 32]>,       // 派生的会话密钥
}
```

存储在 `AppState` 中：

```rust
pub struct AppState {
    pub db: Arc<Mutex<db::Database>>,
    pub identity: Arc<RwLock<identity::IdentityManager>>,
    pub sessions: Arc<RwLock<HashMap<String, SessionState>>>,  // 按对方公钥索引
}
```

### 生命周期

1. 打开聊天窗口 → 生成临时密钥对 → 状态 `None` → `WaitingForPeer`
2. 收到对方 `KeyExchange` → 验证签名和时间戳 → 派生 SessionKey → 状态 `KeyExchanged`
3. 发送 `KeyConfirm` → 状态 `WaitingForConfirm`
4. 收到对方 `KeyConfirm` → 解密确认 → 状态 `Active`
5. 发送/接收消息时检查状态为 `Active`
6. 关闭聊天窗口 → **销毁 SessionState**（清零内存）

## 内存安全

使用 `zeroize` crate 确保敏感数据在销毁时清零：

```toml
# Cargo.toml
zeroize = "1"
```

```rust
use zeroize::Zeroize;

impl Drop for SessionState {
    fn drop(&mut self) {
        self.my_ephemeral_privkey.zeroize();
        if let Some(ref mut key) = self.session_key {
            key.zeroize();
        }
        if let Some(ref mut key) = self.peer_ephemeral_pubkey {
            key.zeroize();
        }
    }
}
```

## 边界情况

### 对方离线

- 临时公钥已发出，对方未回复 KeyExchange
- 消息暂存在内存中的明文队列，状态为 `WaitingForPeer`
- 对方上线后收到 KeyExchange → 派生 SessionKey → 加密后存入数据库 → 发送

### 网络断开

- 重新连接后发起新 KeyExchange，协商新 SessionKey
- 内存中的明文消息用新密钥加密后发送

### 并发 KeyExchange

- 双方同时打开聊天窗口时，双方都会发送 KeyExchange
- 收到对方 KeyExchange 后，如果自己尚未发送，则立即发送
- 如果已建立会话后收到新的 KeyExchange，进入 `Rekeying` 状态重新协商

### 前端交互

- 打开聊天窗口时自动发起 KeyExchange
- SessionKey 未就绪时：发送按钮禁用，显示"正在建立安全通道..."
- 收到对方临时公钥后：输入框启用
- 解密失败时：显示"[解密失败]"，提供"重新建立安全通道"按钮

## 复用已有函数

- `crypto::derive_session_key()` — X25519 ECDH + HKDF 派生会话密钥
- `crypto::encrypt_message()` — AES-256-GCM 加密
- `crypto::decrypt_message()` — AES-256-GCM 解密
- `crypto::generate_device_keypair()` — 生成临时 X25519 密钥对（重命名为 `generate_x25519_keypair()`）
- `crypto::sign_data()` — Ed25519 签名
- `crypto::verify_signature()` — Ed25519 验签（当前未使用，将接入）
