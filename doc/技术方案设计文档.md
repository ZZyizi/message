## 1. 概述

### 1.1 产品定位

本系统是一个：

- **弱中心化** + **端到端加密** + **设备为核心身份**的即时通讯协议与应用系统

核心目标：

- 用户数据归用户设备所有
- 消息传输依赖轻量 Relay（非强中心化服务器）
- 支持群聊 / 私聊 / 多设备同步
- 支持离线消息补发与最终一致性
- 支持图片/视频/文件等媒体内容分布式存储

### 1.2 技术选型

| 层级 | 推荐技术 | 说明 |
|------|---------|------|
| 核心协议 | Rust | 跨平台编译到 WASM / Native，低内存 |
| Desktop | Tauri + Svelte | 内存 20~80MB，比 Electron 轻 10x |
| Mobile | Flutter + Rust FFI | UI 与核心逻辑分离 |
| Web | React/Svelte + Rust WASM | 核心逻辑编译为 WASM |
| Relay | Rust + Axum + WebSocket | 极低资源占用，单 binary |
| 媒体存储 | 用户自维护 OSS/R2 桶 | 服务端不存储媒体 |
| 本地数据库 | SQLite（桌面）/ IndexedDB（Web） | 本地优先存储 |
| 密钥缓存 | Redis（Relay侧）/ 内存（客户端） | 群组密钥临时缓存 |

### 2.1 整体架构

```
┌─────────────────────────────────────────────────────────────┐
│                        Client（设备）                         │
│  ┌─────────────┐  ┌──────────────┐  ┌────────────────────┐ │
│  │  UI Layer   │  │ Rust Core    │  │ Background Daemon  │ │
│  │ Svelte/React│  │ crypto/sync  │  │ (后台守护进程)     │ │
│  └─────────────┘  └──────────────┘  └────────────────────┘ │
│         │               │                    │            │
│         └───────────────┬┴────────────────────┘            │
│                         │                                   │
│              ┌──────────▼──────────┐                        │
│              │   Local Database   │                        │
│              │ SQLite/IndexedDB   │                        │
│              └────────────────────┘                        │
└─────────────────────────────────────────────────────────────┘
                         │
                         │ WebSocket
                         ▼
┌─────────────────────────────────────────────────────────────┐
│                      Relay Server                           │
│  ┌─────────────┐  ┌──────────────┐  ┌────────────────────┐ │
│  │ WS Handler │  │  在线状态    │  │  加密消息缓存(TTL)  │ │
│  │             │  │  Redis Map  │  │  群密钥(Redis)     │ │
│  └─────────────┘  └──────────────┘  └────────────────────┘ │
└─────────────────────────────────────────────────────────────┘
                         │
                         │ (Relay 与 Relay 之间无协调)
                         ▼
              ┌──────────────────────┐
              │    用户配置 Relay     │
              │    列表手动切换       │
              └──────────────────────┘
```

### 2.2 数据流概述

1. **消息发送**：客户端加密 → Relay 转发 → 目标设备接收 → ACK 确认
2. **离线同步**：后台守护进程监听 → 目标上线 → 拉取 pending 消息 → 分页同步
3. **媒体上传**：客户端加密 → 上传至用户 OSS/R2 桶 → 广播 media_id
4. **群密钥管理**：Redis 缓存群密钥 → 成员变化时更新 → 被踢成员获取时被拒绝

## 3. 密码学设计

### 3.1 加密体系

| 用途 | 算法 | 说明 |
|------|------|------|
| 密钥交换 | X25519 | Curve25519 椭圆曲线 Diffie-Hellman |
| 消息加密 | AES-256-GCM | 带认证的同步加密 |
| 数字签名 | Ed25519 | 消息不可篡改，身份不可伪造 |
| 哈希 | BLAKE3 | 用于 media_id 计算 |

### 3.2 密钥层次

```
Identity Key（主身份）
  └── 永久保存，用于身份证明
  └── 不参与日常加密，仅用于设备授权签名

Device Key（设备身份）
  └── 由主设备生成，绑定到具体设备
  └── 用于日常消息加密签名

Session Key（会话密钥）
  └── 由 X25519 动态生成
  └── 用于单次会话或单次消息加密

Group Key（群组密钥）
  └── 由群主生成，存储在 Redis/内存
  └── 所有群成员共享
  └── 成员变化时轮转
```

### 3.3 加密流程

**私聊消息加密流程：**

```
1. 发送方获取接收方 Device Key（首次从 Relay 拉取，之后本地缓存）
2. 使用 X25519 协商 Session Key
3. 使用 AES-256-GCM 加密消息
4. 使用 Ed25519 签名
5. 发送：from + to + timestamp + payload(加密数据) + signature
```

**群聊消息加密流程：**

```
1. 发送方从 Redis/内存获取 Group Key
2. 使用 AES-256-GCM 加密消息
3. 使用 Ed25519 签名
4. Relay 仅转发加密数据，不解密
```

### 3.4 群密钥管理（边界问题 #2 确认）

- **存储位置**：Redis（Relay 侧）+ 发送方内存
- **获取控制**：成员被踢出后，下次获取群密钥时被 Relay 拒绝（返回 403）
- **密钥轮转触发时机**：
  - 成员被踢出时触发轮转
  - 群主主动触发
- **历史消息处理**：历史消息保持旧密钥，被踢成员无法解密后续消息

## 4. 消息系统

### 4.1 消息结构

```json
{
  "event_id": "uuid-v4",
  "type": "chat.message",
  "from": "device_pubkey",
  "to": "recipient_pubkey | group_id",
  "timestamp": 1748160000000,
  "payload": "base64(AES-256-GCM(plaintext))",
  "media_id": "optional_sha256_hash",
  "signature": "ed25519_signature",
  "seq_id": 12345
}
```

### 4.2 ACK 机制

```
1. 接收方收到消息后，发送 ACK 事件
2. 发送方收到 ACK 后，从 pending queue 移除
3. 未确认消息进入 pending queue
4. 设备上线后自动补发 pending queue 中的消息
```

**ACK 事件结构：**

```json
{
  "event_id": "ack_uuid",
  "type": "message.ack",
  "ref_event_id": "原消息event_id",
  "from": "device_pubkey",
  "timestamp": 1748160000001
}
```

### 4.3 离线消息补发（边界问题 #1 #7 确认）

**冲突处理策略：**

```
1. 每条消息携带 seq_id（全局递增序号）
2. 同一 seq_id 被多次提交时：
   - 保留 timestamp 较早的消息
   - 后续消息标记为"冲突消息"，展示给用户
3. 冲突时按发送方 pubkey 优先级排序（群主 > 管理员 > 普通成员 > 其他）
```

**Pending Queue 限制（边界问题 #7 确认）：**

- **时间限制**：仅保留 7 天内未确认消息
- **分页获取**：每次同步最多拉取 100 条
- **群主配置**：群主/管理员可配置保留天数（1~30 天）
- **超出处理**：超出时间窗口的消息标记为"已过期"，不补发

**补发流程：**

```
1. 设备上线 → 连接 Relay
2. 拉取 pending queue（分页）
3. 按 seq_id 排序
4. 处理冲突（如有）
5. 逐步展示消息
6. 发送 ACK 确认
7. 后台守护进程持续监听新消息
```

### 4.4 消息撤回（边界问题 #6 确认）

**撤回事件结构：**

```json
{
  "event_id": "recall_uuid",
  "type": "message.recall",
  "ref_event_id": "原消息event_id",
  "from": "device_pubkey",
  "timestamp": 1748160000002
}
```

**同步机制：**

```
1. 用户发起撤回 → 发送 recall 事件
2. 已上线设备：Relay 立即广播 recall 事件
3. 离线设备：后台守护进程在同步时发现撤回事件 → 执行撤回
4. 目标收到 recall 后：
   - 删除本地消息（保留撤回标记）
   - 媒体文件保留（无法真正删除 OSS 中的数据）
```

## 5. 群聊系统

### 5.1 群结构

```json
{
  "group_id": "uuid",
  "owner": "pubkey",
  "admins": ["pubkey1", "pubkey2"],
  "members": ["pubkey3", "pubkey4"],
  "created_at": 1748160000000,
  "group_key_id": "current_key_id",
  "metadata": {
    "name": "群名称",
    "avatar": "media_id"
  }
}
```

### 5.2 群密钥存储

```
Redis Key: group_key:{group_id}:{key_id}
Value: AES-256-GCM 加密的群密钥
TTL: 根据群主配置（默认 30 天）

客户端内存缓存：
- 当前使用的 group_key
- 历史 group_key（用于解密历史消息，限时解密）
```

### 5.3 成员变化事件

```json
{
  "event_id": "uuid",
  "type": "group.member_added | group.member_removed",
  "group_id": "uuid",
  "operator": "pubkey",
  "target": "pubkey",
  "timestamp": 1748160000000,
  "new_group_key_id": "可选，密钥轮转时携带"
}
```

### 5.4 同步优先级

```
1. 群主设备（最高优先级）
2. 管理员设备
3. 最近在线成员
4. Relay 缓存（fallback）
```

### 5.5 分片同步策略

```
sync_range: [start_seq_id, end_seq_id]

- 避免全量拉取
- 支持断点续传
- 按时间区间或 seq_id 区间同步
```

## 6. 多设备系统

### 6.1 身份结构

```
Identity Key（主身份）
  └── 永久存储在主设备
  └── 用于授权 Device Key

Device Key（设备身份）
  └── 每个设备独立生成
  └── 绑定到具体设备
```

### 6.2 设备绑定流程（边界问题 #4 确认）

**主设备生成授权 Token：**

```
1. 主设备生成一次性授权 token（UUID + 有效期 5 分钟）
2. 展示为二维码（二维码内容含 token + 主设备公钥）
```

**新设备扫码绑定：**

```
1. 新设备扫描二维码
2. 新设备生成自己的 Device Key
3. 新设备使用 token 请求主设备授权
4. 主设备确认后：
   - 签名新设备的公钥
   - 返回授权证书
5. 新设备保存授权证书
6. 同步历史数据（通过安全信道）
```

**授权证书结构：**

```json
{
  "cert_id": "uuid",
  "device_pubkey": "新设备公钥",
  "identity_pubkey": "主身份公钥",
  "permissions": ["read", "write", "sync"],
  "expires_at": 1748163600000,
  "signature": "主身份私钥签名"
}
```

### 6.3 设备撤销

```
1. 主设备发起撤销请求
2. Relay 更新黑名单
3. 被撤销设备无法再连接（返回 403）
4. 广播设备变更事件
```

### 6.4 设备同步策略

```
P2P 优先同步（同一局域网）
Relay 辅助同步
分片历史同步
```

## 7. 媒体系统

### 7.1 设计原则（边界问题 #5 确认）

- **媒体不走消息流**
- **媒体独立存储于用户 OSS/R2 桶**
- **服务端不存储媒体数据**
- **消息仅引用 media_id**

### 7.2 媒体结构

```json
{
  "media_id": "sha256(file)",
  "mime": "image/jpeg",
  "size": 12345,
  "encryption_key": "base64(AES-256-GCM key)",
  "bucket": "user_bucket_name",
  "region": "oss-cn-hangzhou",
  "endpoint": "https://oss.example.com"
}
```

### 7.3 上传流程

```
1. 客户端加密文件（AES-256-GCM）
2. 计算 hash 作为 media_id
3. 获取用户 OSS/R2 配置（从本地配置读取）
4. 直接上传到用户 OSS/R2 桶（不经 Relay）
5. 构造 media_id 事件，发送到 Relay
6. Relay 广播 media 引用事件
```

### 7.4 下载流程

```
1. 收到消息，含 media_id
2. 查询多个节点（Relay 缓存 / 群成员 / 发送者）
3. 下载加密 blob
4. 使用消息中的 encryption_key 本地解密
5. 展示给用户
```

### 7.5 OSS/R2 配置管理

```
配置存储位置：客户端本地配置
配置内容：
- bucket 名称
- endpoint
- access_key_id
- access_key_secret（加密存储）

用户自行维护自己的 OSS/R2 桶，服务端不存储任何媒体。
```

## 8. 同步机制

### 8.1 同步触发时机

| 场景 | 触发方式 |
|------|---------|
| 应用启动 | 主动拉取 |
| 连接 Relay | 增量同步（基于 last_seq_id） |
| 后台守护进程 | 增量轮询（心跳 30s） |
| 收到 push | 立即处理 |

### 8.2 同步优先级（边界问题 #7 确认）

```
1. 未确认消息（pending queue）
2. 撤回事件（message.recall）
3. 群组成员变化
4. 群组密钥变化
5. 新消息
```

### 8.3 分页同步协议

```json
{
  "sync_type": "pagination",
  "last_seq_id": 12340,
  "limit": 100,
  "direction": "forward"
}
```

### 8.4 同步冲突处理

```
冲突检测：
- 同一 seq_id 被多次接收
- timestamp 相同但内容不同

冲突解决：
- 保留较早 timestamp 的消息
- 较晚消息标记为"疑似重复"
- 显示给用户，用户可手动处理
```

## 9. Relay 服务器

### 9.1 职责范围

**Relay 负责：**

- WebSocket 连接管理
- 在线用户映射（Redis）
- 消息转发
- 加密消息临时缓存（TTL，默认 7 天）
- 群事件转发
- 群组密钥缓存（Redis）
- 设备黑名单

**Relay 不负责：**

- 解密消息
- 用户关系存储
- 长期数据存储
- 媒体存储

### 9.2 Relay 高可用（边界问题 #3 确认）

- **无 Relay 间协调**
- 用户手动配置 Relay 列表
- 切换策略：优先选择延迟最低的 Relay
- 典型配置：

```json
{
  "relay_list": [
    "wss://relay1.example.com",
    "wss://relay2.example.com",
    "wss://relay3.example.com"
  ],
  "auto_switch": true
}
```

### 9.3 在线状态管理

```
Redis Key: online:{pubkey}
Value: {
  "device_id": "xxx",
  "relay_url": "wss://relay1.example.com",
  "last_seen": 1748160000000
}
TTL: 60s（心跳续期）
```

### 9.4 消息缓存

```
Redis Key: msg_cache:{event_id}
Value: 加密消息体
TTL: 7 天（可配置）
```

### 9.5 API 端点

| 路径 | 方法 | 说明 |
|------|------|------|
| /ws | WebSocket | 客户端连接 |
| /api/v1/group_key/:group_id | GET | 获取群密钥（需验证成员身份）|
| /api/v1/group_key/:group_id | POST | 创建/更新群密钥（仅群主）|
| /api/v1/sync | GET | 分页同步消息 |
| /api/v1/online/:pubkey | GET | 查询在线状态 |

## 10. 后台守护进程

### 10.1 设计目标（边界问题 #8 确认）

类似微信后台进程：

- **独立于主应用运行**
- **用户退出应用后仍可同步**
- **占用资源极轻**（内存 < 20MB）

### 10.2 功能范围

```
1. 维护 WebSocket 长连接
2. 增量同步未读消息
3. 处理撤回事件
4. 处理群组成员变化
5. 处理群密钥更新
6. 发送消息确认（ACK）
```

### 10.3 后台进程与主应用的关系

```
┌─────────────────────────────────────┐
│         主应用进程                    │
│  ┌─────────────────────────────┐   │
│  │     UI + 业务逻辑            │   │
│  │     调用守护进程 API          │   │
│  └─────────────────────────────┘   │
└─────────────────────────────────────┘
              │ IPC（Unix Socket / 命名管道）
              ▼
┌─────────────────────────────────────┐
│       后台守护进程（Daemon）          │
│  ┌─────────────────────────────┐   │
│  │   同步引擎 + 加密 + Relay连接 │   │
│  │   独立运行，不依赖主应用       │   │
│  └─────────────────────────────┘   │
└─────────────────────────────────────┘
```

### 10.4 进程间通信

```
主应用 → 守护进程：
- 发送消息请求
- 读取本地数据库
- 获取配置

守护进程 → 主应用：
- 新消息通知
- 在线状态变化
- 同步进度
```

### 10.5 唤醒机制

```
iOS: Background App Refresh / PushKit
Android: Foreground Service + WorkManager
Windows: Windows Service / 后台任务
macOS: LaunchAgent
```

### 10.6 数据共享

```
守护进程与主应用共享：
- 本地数据库文件（SQLite）
- 密钥材料（通过安全存储）
- 用户配置

隔离：
- 各自独立的进程内存
- 消息队列用于 IPC
```

## 11. 安全设计

### 11.1 安全目标

- **Relay 不可解密数据**：Relay 仅转发加密数据
- **消息不可篡改**：Ed25519 签名验证
- **身份不可伪造**：公私钥体系 + 设备授权链

### 11.2 密钥安全存储

```
桌面端：
- 使用系统密钥链（Keychain / KWallet）
- 敏感数据加密存储

移动端：
- iOS: Keychain
- Android: Keystore

Web 端：
- IndexedDB（加密后存储）
- Web Crypto API
```

### 11.3 中间人攻击防护

```
1. 公钥验证（首次连接时确认 fingerprint）
2. 签名验证（每条消息带 Ed25519 签名）
3. Relay 不持有任何可解密密钥
```

### 11.4 隐私保护

```
1. 无需手机号注册
2. 无需邮箱注册
3. 仅需保存 pubkey
4. Relay 不存储用户关系链
```

## 12. 部署架构
